Primary issue

One detail page can rebuild the whole tracking library.

The expensive part is not opening a page by itself. The expensive part is the hidden background metadata refresh that can jump from one tracked title into full-user aggregate repair.

Does it make sense?

Yes. The dashboard spike matches the code path. The nuance is that the rebuild happens when metadata refresh is not throttled, which is likely after importing old records into the new US database.

The Accidental Runtime Path

The first two steps are normal. The fourth step is the cost bug: one show refresh becomes a whole-library tracking rebuild.

Step 01

Tracked detail page opens

app/show/[id].tsx sees authenticated tracking state and a watchlisted title.

Step 02

Background metadata refresh fires

The effect calls refreshTrackedShowMetadata(showLookupArgs).

Step 03

Backend resolves one show

refreshTrackedShowMetadata calls the shared metadata helper.

Step 04

repairUserId turns on broad repair

The helper calls rebuildUserShowTrackingAggregatesForUser.

Step 05

Every tracked row is touched

The batch reads user shows, show docs, watched episodes, then patches aggregates and feed projections.

Why The Cost Lands In Database I/O

Function calls and compute are small. The dominant cost is documents and payloads read from Convex while recomputing derived state.

Current hidden repair path

  • 1 Reads a page of userShows for the current user.
  • 2 Reads each linked shows document.
  • 3 Collects all watchedEpisodes for every user/show pair.
  • 4 Patches userShows and upserts feedProjections.

What normal metadata refresh should do

  • 1 Refresh provider metadata for the one show.
  • 2 Patch the one shows document if metadata changed.
  • 3 Run projection refresh for users tracking that show.
  • 4 Leave aggregate repair to explicit user actions.

The Fix

Keep metadata refresh, remove the accidental aggregate repair. Manual repair remains available from the existing targeted paths.

Change the routine show-detail path to opt out of broad repair by passing skipBroadAggregateRepair: true.
