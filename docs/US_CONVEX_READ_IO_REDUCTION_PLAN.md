# US Convex Read I/O Reduction Plan

Last updated: 2026-05-16

## Goal

Reduce the recurring Convex DB I/O on the new US deployment without changing what users see in Home, Watchlist, Upcoming, Discover, Recommendations, or Library.

The abnormal backfill path is handled separately in `docs/TRACKED_METADATA_REFRESH_BACKFILL_FIX_PLAN.md`. This plan covers the remaining recurring read-heavy paths:

- `schedule.getFutureUpcomingCountsForWatchlist`
- `schedule.getUpcomingSchedule`
- `shows.getTrackedIds`
- `shows.getLibrary`

## Important Repo Rule

`AGENTS.md` marks watchlist, schedule, schedule cache, episode availability, provider matching, dedupe, projections, and reconciler boundaries as critical. Any implementation that affects those areas requires a new ADR before or in the same PR as the code change.

This planning file is not the ADR. The schedule projection implementation PR should add a new ADR using the next unused number, for example:

```text
docs/ADR-0005-user-schedule-projections.md
```

The ADR must cover behavior, provider assumptions, title fallback rules, same-day dedupe, completed-show reactivation, stale provider totals, verification, and rollback notes.

## Answer To "Can We Move This To The Backend?"

Yes, but with one important caveat: these functions already run on the Convex backend. Moving the same range scan from a reactive query into an action would not remove the database I/O. It would only move when the I/O happens.

The useful move is:

```text
repeated reactive query scan
  -> backend projection/reconciliation job
  -> compact per-user schedule/count rows
  -> cheap reactive query reads
```

That keeps the same UI functionality while avoiding repeated scans of global `scheduleCache` rows for every Home or Upcoming subscription.

Longer term, this also lines up with `docs/SCHEDULE_CONFIDENCE_PLAN.md`: provider-wide schedule/release intelligence should live in the external reconciliation backend, and Convex should receive compact deltas/projections instead of repeatedly doing broad provider/schedule scans in app queries.

## Current Cost Shape

### 1. `schedule.getFutureUpcomingCountsForWatchlist`

Current caller:

- `app/(tabs)/home/index.tsx`
- Active when `activeTab === "watchlist"`
- Uses the watchlist future window, currently a 90-day lookahead.

Current backend shape:

- Reads all TV/anime `feedProjections` for the user.
- Scans `scheduleCache` rows by date range.
- Parses global schedule JSON payloads.
- Matches global entries back to the user's tracked rows.
- Computes `availableCount`, `futureCount`, and `unavailableCount` per route.

Why it costs I/O:

- `scheduleCache` is global, not user-specific.
- Each call reads schedule rows for all shows in the date range, even though the result only needs rows for this user's tracked shows.
- Dashboard showed about 93.81 MB across 153 calls, about 613 KB per uncached call.

### 2. `schedule.getUpcomingSchedule`

Current caller:

- `app/(tabs)/home/index.tsx`
- Active only when `activeTab === "upcoming"`
- Reads either compact week-ish ranges or the visible month grid range.

Current backend shape:

- Reads TV/anime `feedProjections` for the user.
- Scans `scheduleCache` rows by date range.
- Parses global schedule JSON payloads.
- Matches and dedupes against tracked shows.
- Groups by date for the Upcoming UI.

Why it costs I/O:

- It has the same global schedule-cache scan shape as the future-count query.
- The I/O is lower because it runs on the Upcoming tab instead of Home Watchlist, but it still scales with global cache payload size rather than the user's tracked schedule.

### 3. `shows.getTrackedIds`

Current callers:

- Discover
- Recommendations

Current backend shape:

- Reads up to 1000 `userShows`.
- For each row, reads the linked `shows` document.
- Builds the tracked-status map used by UI cards.

Why it costs I/O:

- This is an N+1 hydration pattern.
- It is visible in dashboard I/O and logs because the call can take around a second for a larger library.

### 4. `shows.getLibrary`

Current backend shape:

- Reads matching `userShows`.
- For each row, reads the linked `shows` document.
- Returns hydrated library entries.

Why it costs I/O:

- Same broad-library plus N+1 pattern.
- It is not the biggest current cost, but it gets worse as the library grows.

## Proposed Architecture

### A. Add User-Specific Schedule Projections

Add a compact user-specific projection table for schedule events, generated by backend reconciliation code instead of by every reactive query.

Working name:

```text
userScheduleEvents
```

Likely fields:

- `userId`
- `date`
- `routeId`
- `showId`
- `feedProjectionId`
- `mediaType`
- `sourceMediaType`
- `showTitle`
- `posterUrl`
- provider IDs needed for routing and matching: `tmdbId`, `anilistId`, `malId`, `tvmazeId`, `imdbId`
- episode fields: `seasonNumber`, `episodeNumber`, `episodeName`, `airDate`
- sort/dedupe fields: `airtimeMs`, `seriesDedupeKey`, `episodeDedupeKey`, `sameTrackedShowDayKey`
- match/audit fields: `matchSource`, `sourceMatchesTracked`, `matchConfidence`
- source freshness fields: `scheduleCacheUpdatedAt`, `projectionUpdatedAt`

Likely indexes:

```ts
.index("by_user_date", ["userId", "date"])
.index("by_user_date_media", ["userId", "date", "mediaType"])
.index("by_user_route_date", ["userId", "routeId", "date"])
.index("by_user_projection_date", ["userId", "feedProjectionId", "date"])
```

The exact schema should be chosen during implementation after checking generated Convex types and existing helper contracts.

Before finalizing the table name, inspect the US deployment for any existing or orphaned schedule projection table from earlier experiments. If one exists but is not represented in `convex/schema.ts`, decide explicitly in the ADR whether to adopt it, migrate it, or leave it untouched. Do not delete production data as part of this optimization without a separate migration decision.

### B. Add Compact Future Count Projections

`getFutureUpcomingCountsForWatchlist` returns count rows, not schedule rows. To make that query cheap even when called often, add a compact count projection for the exact Home watchlist window.

Working name:

```text
watchlistFutureCountProjections
```

Likely fields:

- `userId`
- `windowStartDate`
- `windowEndDate`
- `mediaType`
- `routeId`
- `availableCount`
- `futureCount`
- `unavailableCount`
- `projectionUpdatedAt`

Likely indexes:

```ts
.index("by_user_window", ["userId", "windowStartDate", "windowEndDate"])
.index("by_user_window_media", ["userId", "windowStartDate", "windowEndDate", "mediaType"])
```

This preserves the current 90-day Home behavior. If the UI later needs arbitrary custom windows, either materialize those windows intentionally or read from `userScheduleEvents` as the fallback.

### C. Build Projections Outside The Reactive Read Path

Create an internal backend refresh path that:

1. Reads the user's TV/anime `feedProjections`.
2. Reads `scheduleCache` for the requested date window.
3. Uses the existing schedule matching and dedupe helpers:
   - `findTrackedScheduleMatch`
   - `shouldCollapseSameTrackedShowDay`
   - `shouldPreferScheduleCandidate`
   - `shouldPreferScheduleEpisode`
   - `shouldPreferSameTrackedShowDayEpisode`
4. Writes compact `userScheduleEvents`.
5. Writes compact `watchlistFutureCountProjections`.
6. Records refresh state so the same user/window is not rebuilt repeatedly.

The refresh should be idempotent and chunked. It should patch/insert stable projection keys and remove stale rows for the refreshed window. Avoid rebuilding the full user schedule synchronously inside a query.

Trigger options:

- Short term: piggyback on the existing Home schedule signal refresh flow, likely near `syncHomeCachedScheduleSignals`, but keep the app query itself read-only and cheap.
- On track/untrack/status changes: mark the user's schedule projection dirty, then refresh asynchronously.
- On schedule-cache or schedule-confidence imports: mark affected projection windows dirty or apply compact deltas.
- Long term: have the external SQLite schedule-confidence backend produce compact schedule/release deltas for Convex, then Convex only updates user-facing projection rows.

### D. Change Read Queries To Read Projections

After projections exist:

- `schedule.getUpcomingSchedule` should read `userScheduleEvents` by `userId + date range`, apply media filtering, group by date, and sort by airtime/title.
- `schedule.getFutureUpcomingCountsForWatchlist` should read `watchlistFutureCountProjections` for the current window and media filter.
- `schedule.getTodayScheduledWatchlistFeed` can later read `userScheduleEvents` for today's date instead of scanning `scheduleCache`.

During rollout, keep the old `scheduleCache` scan as a temporary fallback behind a narrow guard, then remove it after the US deployment has enough projection coverage and dashboard numbers confirm the new path is stable.

## Low-Risk Optimizations First

These are separate from the schedule projection work and can be implemented first.

### 1. Replace `shows.getTrackedIds` N+1 Reads

Use `feedProjections` as the source for tracked IDs instead of `userShows` plus per-show hydration.

Why this should preserve functionality:

- `feedProjections` already stores `mediaType`, `tmdbId`, `anilistId`, `malId`, tracking `status`, watched count, and `totalEpisodes`.
- Those are the fields needed to build the Discover/Recommendations tracked-status map.

Implementation notes:

- Preserve the existing dedupe preference: completed/watched wins, then higher watched count, then newest update.
- Preserve anime fallback behavior where MAL/Jikan identity is needed.
- Keep the same public return shape unless the UI needs `malId`; if it does, add it deliberately and update callers.

Expected effect:

- Removes up to 1000 show-document reads per call.
- Should noticeably reduce `shows.getTrackedIds` I/O and latency.

### 2. Reduce `shows.getLibrary` Hydration

Audit the Library UI fields before changing behavior.

If Library only needs fields already in `feedProjections`, switch `getLibrary` to read `feedProjections` with existing user/media/status indexes.

If Library still needs fields not currently denormalized, such as `overview`, `genres`, or `rating`, choose one of these:

- Add those fields to `feedProjections` and backfill them.
- Split Library into a compact list query plus detail-level hydration only when a row is opened.

Expected effect:

- Removes N+1 show hydration for the library list.
- Keeps Convex as the source of truth for user-owned tracking state.

## Implementation Order

### PR 1: Cheap tracked/library reads

Scope:

- Refactor `shows.getTrackedIds` to use `feedProjections`.
- Audit and, if safe, refactor `shows.getLibrary` to use `feedProjections`.
- Add or update focused tests/fixtures for anime fallback IDs, completed status, partial progress, and duplicate identities.
- Record the tracked-ID read decision in `docs/ADR-0004-tracked-ids-projection-read.md`.

Why first:

- Lower behavioral risk than schedule projection work.
- Reduces visible recurring I/O immediately.
- Does not require changing schedule matching behavior.

### PR 2: ADR and schedule projection schema

Scope:

- Add the required schedule/watchlist ADR.
- Add projection tables and indexes.
- Add internal mutations for idempotent upsert/delete of projected schedule rows and count rows.
- Add projection state/dirty markers.

No UI switch yet.

Why separate:

- Lets schema and migration mechanics be reviewed without changing Home or Upcoming output.

### PR 3: Projection builder and validation

Scope:

- Add the internal action/query/mutation chain that builds projections from existing `feedProjections` and `scheduleCache`.
- Reuse existing matching/dedupe helpers instead of rewriting matching logic.
- Add fixture-level validation for known risk cases from ADR 0002:
  - same-day duplicates
  - TVMaze TV rows versus anime rows
  - anime aliases
  - completed shows with new releases
  - stale provider totals
  - future episodes versus available episodes

No permanent UI switch yet.

### PR 4: Switch reads to projections

Scope:

- Change `getUpcomingSchedule` to read `userScheduleEvents`.
- Change `getFutureUpcomingCountsForWatchlist` to read `watchlistFutureCountProjections`.
- Keep old schedule-cache scans behind a temporary guarded fallback.
- Use dashboard/log metrics to confirm the fallback is not being hit for normal US production usage.

### PR 5: Remove fallback and extend same-day feed

Scope:

- Remove old fallback scans once projection coverage is proven.
- Optionally change `getTodayScheduledWatchlistFeed` to read today's `userScheduleEvents`.
- Update the ADR if the final behavior differs from the original projection design.

## Rollout And Verification

Minimum local checks:

```bash
npx expo lint
npx tsc --noEmit --pretty false
git diff --check
```

Convex checks:

```bash
npx convex deploy --dry-run --yes
```

Do not start or restart the user's frontend/backend servers unless explicitly requested.

Behavior checks before merging schedule changes:

- Home Watchlist active rows still match current behavior.
- Home Watchlist future-only hiding still works.
- Upcoming tab shows the same selected-day and month/week rows as before.
- Media filters still match current behavior.
- Route IDs still open the same show pages.
- Same-day duplicate provider rows still collapse correctly.
- Completed shows with new available episodes still re-enter attention.
- Paused, dropped, and planned rows keep their current schedule/watchlist semantics.

Operational checks after deploying to the US project:

- `schedule.getFutureUpcomingCountsForWatchlist` I/O should drop from global range-scan size to compact count-row size.
- `schedule.getUpcomingSchedule` I/O should scale with this user's projected events, not global schedule cache size.
- `shows.getTrackedIds` should no longer show N+1 latency or high read volume.
- `shows.getLibrary` should stop growing linearly with both `userShows` and `shows` hydration reads.

## Things Not To Do

- Do not simply move the existing schedule scans into an action and call it from the UI. That changes timing, not I/O volume.
- Do not narrow the 90-day future window as a cost fix unless the product behavior is intentionally changed.
- Do not broaden title fallback to make projection generation "find more matches." That can reintroduce duplicate and wrong-row bugs.
- Do not delete `scheduleCache` or remove old query fallbacks until projection coverage is verified.
- Do not store image files in Convex; projection rows should only store external image URLs.
- Do not bypass Convex for user-owned state. External backend/reconciler work should send compact deltas back into Convex.

## Expected Priority

1. `shows.getTrackedIds`: quick win, low risk, removes obvious N+1 reads.
2. `schedule.getFutureUpcomingCountsForWatchlist`: biggest recurring normal-app I/O, needs ADR/projection work.
3. `schedule.getUpcomingSchedule`: same projection model as future counts, lower call frequency.
4. `shows.getLibrary`: optimize after auditing UI fields, unless dashboard shows it climbing.
